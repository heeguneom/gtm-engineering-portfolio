# Changelog — Link Suggestion Tool spec

## 2026-07-20

- Spec opened. Original seed was "fuse RAG semantic search with the Obsidian knowledge graph" (GraphRAG pattern) over `~/.claude/reports`.
- **Pivot before scaffolding:** investigated actual `[[wiki-link]]` density in the vault before assuming a graph exists to fuse with. Finding: only 4 of 561 files contain `[[...]]` syntax at all, and only 2 are genuine cross-document links (the rest are documentation examples showing link syntax, not real links). The premise "the graph that already exists" was false for this vault.
- Presented three paths to user: (A) build fusion anyway despite near-empty graph, (B) invert the project — use existing RAG/embedding infrastructure to *suggest* links and help build a real graph, (C) do nothing further, RAG alone already works.
- User chose **B**. Spec re-scoped from "GraphRAG fusion" to "RAG-assisted link suggestion tool."
- Dispatched `/worldmodel` — confirmed no link-suggestion capability exists anywhere in the toolchain (verified `obsidian-mcp`'s actual installed README, 12 tools, no graph/backlinks). Surveyed prior art: Smart Connections, Semantic Backlinks, Semantic Auto-Linker, Similar Notes. Strong convergence on "suggest-only, human review" pattern; no industry-standard threshold; chunk-vs-file-level aggregation is an unresolved fork industry-wide, not just here. Persisted 3 evidence files.
- Presented 4-decision batch (trigger shape, aggregation strategy, threshold, persistence). User confirmed all 4 recommendations (A/A/B/A) in one pass. Logged as D1 (LOCKED, 1-way door — new tool surface), D2 (DIRECTED, max-pool aggregation), D3 (DIRECTED, 0.5 threshold), D4 (LOCKED, in-memory computation, napkin-math-backed).
- Cascaded D1-D4 into §5 (journeys), §6 (requirements), §9 (proposed solution), §11 (all P0 open questions resolved), §13 (In Scope, passed resolution completeness gate).
- **Completeness re-sweep caught 3 real gaps** not surfaced in the original decision batch: same-file chunk-pair exclusion (a file would trivially "match itself"), symmetric-pair deduplication, and wiki-link syntax-variant coverage (`[[Target|Display]]`, `[[Target#Heading]]` — the vault's own docs contain examples of both, and Q6's parser requirement was originally under-specified). Added as three new Must requirements in §6.
- **Audit + design-challenge (parallel subprocesses) ran against the spec.** 7 design-challenge findings (2 High, 3 Medium, 2 Low) + 3 audit findings (2 High, 1 Low). Applied 5 low-cost corrections directly (template-similarity risk named, "no new tool" alternative documented, journey wording tightened to per-suggestion confirmation, unevidenced growth clause replaced with the base spec's confirmed growth rate, minor stderr-precedent wording fix).
- **One real decision reopen, evaluated and resolved with the user:** audit found D4's napkin math was stale (assumed ~1,800 chunks; actual measured 11,536 via `np.load('index_data/vectors.npy').shape`) and flagged an O(n²) vs O(n) complexity mismatch. Independently re-verified by direct measurement on the real index: chunk-level all-pairs = 532MB / 0.35s (not 13MB as originally claimed, though time budget technically still held). Combined with the design-challenger's independent Finding 1 (file-centroid alternative wasn't evaluated against max-pool), and my own direct verification that 73/561 vault files share a basename (53 named `REPORT.md`) confirming the challenger's Finding 2 (link-resolution ambiguity) was a real, non-edge-case gap — presented one consolidated reopen to the user. User confirmed the revised design.
- **D2 superseded:** chunk-pair max-pool → file-level centroid aggregation (mean-pool existing chunk vectors per file, compare file-to-file: 561² vs 11,536² comparisons, 423x cheaper, measured). D4 corrected (napkin math fixed, conclusion strengthened — file-level is cheaper than the original chunk-level estimate even was). Two new decisions logged: D5 (parse links from already-loaded index text, not disk) and D6 (fail-open basename resolution for wiki-link targets). Cascaded into §5, §6, §9, §10, §14, §15. Chunk-level max-pool moved to Future Work (Explored tier) as the accepted-trade-off escalation path.
- **Finalization (Step 8):** fixed remaining stale references to the superseded chunk-level design (§6 performance NFR, NG3's trigger wording — updated to reflect the two-tier centroid→chunk-max-pool escalation path). Ran mechanical pre-finalization checks: no ASSUMED decisions load-bearing, no LOW/MEDIUM-confidence assumptions underpin the one 1-way door (D1, HIGH confidence, explicitly confirmed), non-goal temporal tags verified correct. Derived Agent Constraints (§16) from the finalized In Scope item. Re-verified baseline commit unchanged (`0962921` — spec-only changes, no code touched yet, as designed). Status: Draft → Approved.
- **Implementation:** built `linking.py` (centroid aggregation, index-text link parsing, fail-open resolution, self/symmetric-pair exclusion), registered `suggest_links` in `server.py`, wrote `tests/test_linking.py` (11 tests) mirroring existing suite style. 32/32 tests pass (31 existing + new), `ruff check` clean, zero changes to `chunking.py`/`indexer.py` (SCOPE respected).
- **Live validation run #1** (real 11,536-chunk index, top_k=15, threshold=0.5): 0.129s. Top result (1.000): `bdr-onboarding-playbook/onboarding-blueprint.md` ↔ `bdr-team-onboarding-blueprint/REPORT.md` — a genuine, previously-unlinked match, validating G1. But 14/15 results were tailored-resume pairs (0.97-1.0) — the §14 template-similarity risk confirmed real, not theoretical.
- **D7 added:** exclude `Resume-`-pattern files from consideration (95/561 files, verified 100% consistent naming, precise not heuristic). Cascaded into `linking.py` (`_TEMPLATE_EXCLUDE_RE`, applied in `_group_by_file`), new test `test_excludes_template_files_from_both_sides_of_a_pair`, Decision Log.
- **Live validation run #2** (post-D7, same params): 0.107s. Resume noise gone. BDR pair still #1 at 1.000. 5-6 of remaining 15 are genuine topical matches (Gumloop interview prep/report, unify audit docs, GTM playbook pair, BDR-KPI framework/summary). A second, smaller template cluster observed (`APPLY-TODAY.md` daily job-queues, numbered glossary pages) — left unexcluded, surfaced to user rather than silently patched further. Awaiting user decision on whether to extend D7 or accept as-is.
- **First real graph-building pass (2026-07-20).** Pulled top-60 suggestions (threshold 0.5) instead of just top-15/20, surfacing a new high-confidence pattern: `REPORT.md` files linking to their own `evidence/*.md` files. User approved 14 pairs spanning BDR onboarding, Gumloop interview docs, Unify audits, GTM playbooks, BDR-KPI docs, 3 REPORT↔evidence pairs, SEO content/strategy docs, two dated performance reports, and AI-progression REPORT↔evidence (2 links). Explicitly excluded: the `APPLY-TODAY`/`OUTREACH` daily-queue cluster and the glossary/audit-findings numbered series (same template-noise mechanism as resumes, D7). Inserted via a one-off script (existence-verified against all 26 target files first; idempotent — skips already-present links). **Result: vault link density went from 2 genuine cross-document links (out of 561 files) to 28 new links across 26 files (30 files now have at least one link, 48 total `[[...]]` occurrences vault-wide).** This is the first evidence the tool actually accomplishes G1 in practice, not just in a live-validation dry run.
- **Real bug found and fixed, post-insertion.** Reindexed to pick up the 26 new files (11,536 → 11,610 chunks), then re-ran `suggest_links` to confirm the just-linked BDR pair was now excluded. It wasn't — still the #1 result. Root cause: `_resolve_target` (D6) only ever checked a link target against each file's basename/stem, never the full relative path. But the links just inserted use full-path targets (`[[bdr-team-onboarding-blueprint/REPORT]]`), specifically *because* bare basenames are ambiguous for the 53 files named `REPORT.md` — the exact case D6 was written to handle correctly. Fixed `_resolve_target` to try a full-path match first (case-insensitive, `.md` optional), falling back to basename-only for bare-title links like the vault's original `[[APPLICATIONS]]`. Added regression test `test_recognizes_full_path_links_not_just_bare_basenames`. Re-verified on real data: BDR pair now correctly excluded; all 33 tests pass; `ruff` clean.
- **Built `weekly-link-digest.sh`** + `com.heegun.reports-link-digest` launchd job (weekly, matching the base project's documented growth rate) — reindexes, calls `suggest_links(top_k=40)`, filters the known template-noise clusters (APPLY-TODAY/OUTREACH daily queues, glossary/audit-findings series — same judgment applied manually during the graph-building pass above), dedupes against a running "already surfaced" log, writes a reviewable digest. Suggest-only — never inserts a link itself, matching G3.
